  • 浏览器缓存

    • disk cache，存储在硬盘中下次进入还存在

    • memory cache，进程结束后销毁，存储在内存中

    • 缓存机制

      • 强缓存

        • 由浏览器本地判断缓存的有效性，有效则200 from cache
        • 服务器返回三种headerPragma,Cache-Control[毫秒],Expires[时间戳]后两者都是与计算机本地时间比较

      • 协商缓存

        • 强缓存失效后向浏览器请求询问缓存有效性，有效则304 Not Modified
        • 两种header Last-Modified[时间]/ETag[标识字符串]
        • 资源首次请求成功后服务器会在资源res头上加入Last-Modified/ETag
        • 协商缓存时在请求头中将资源的Last-Modified/ETag作为If-Not-Modified和If-Not-Match的值
        • 服务器判断

      • 优先级

        pragma[http1.0]->catch-control[http1.1]毫秒->expires[http1.0]时间戳->过期->同时存在时Etag->Last-Modified

      • 刷新行为

        1. f5会跳过强缓存进入协商缓存
        2. ctrl-f5设置所有缓存都失效请求全新数据

      • 其他

        • Cache-Control字段:

          Cache-Control: max-age //存储的最大时长
Cache-control: s-maxage //覆盖max-age或者Expires头，只可以用于public资源
Cache-control: no-cache //强制协商缓存验证
Cache-control: no-store //不缓存
Cache-control: public //响应可以被任何对象（包括：发送请求的客户端，代理服务器，等等）缓存
Cache-control: private //只有用户本地浏览器可以缓存

        • 为什么要两种协商缓存

          1. Last-Modified标注的最后修改只能精确到秒级，如果某些文件在1秒钟以内，被修改多次的话，它将不能准确标注文件的修改时间

          2. 如果某些文件会被定期生成，当有时内容并没有任何变化，但Last-Modified却改变了，导致

          3. 有可能存在服务器没有准确获取文件修改时间，或者与代理服务器时间不一致等情形

    • webpack中hash

      • hash

        是工程级别的，即每次修改任何一个文件，所有文件名的hash至都将改变。所以一旦修改了任何一个文件，整个项目的文件缓存都将失效。

      • content hash

        是针对文件内容级别的，只有你自己模块的内容变了，那么hash值才改变
